> 状态：可执行设计（吸收 Pass 工程经验 + 本仓库 `storage.sync` 约束 + 《同步设计参考》合理部分）  
> 基线代码：19.0（整包 `lastUpdated` LWW + `sanitizeForSync` + 单条 7500B 阈值）  
> **实现进度（已合入代码）：** Phase 0–2 主体：`sync_policy/ids/projection/merge/bundle/pack/outbox/engine`；persist 本地优先 + schedulePush；onChanged/init pull；设置页体积/立即同步/同步包；popup 只写 local。Phase 3 已做。**自建 HTTP JSON 同步（Phase 4 子集）已做：** `scripts/sync-server.mjs` + `sync_http_transport.js` + 设置项 `syncTransport=http`。

## 0. 一句话方案

**本地完整状态为权威工作副本；同步传「瘦状态投影 + revision」；合并用实体/字段级 LWW + 墓碑 + 成员关系状态；推送失败走 outbox；写前 safety 快照；空远端与丢 ID 一律拒绝。**  
完整 op log / HLC 作为容量与离线顺序要求升高后的 Phase 3+，不阻塞 v1。

## 1. 目标与非目标

### 1.1 目标

| ID | 目标 | 验收 |
|----|------|------|
| G1 | 同浏览器账号下多设备共享分组/书签结构 | 设备 A 增删改后，B 打开新标签或短时内可见 |
| G2 | 双端同时编辑尽量少丢数据 | 不再整库互盖；冲突可解释 |
| G3 | 本机数据在同步失败/超配额时不丢 | local 始终可读写 |
| G4 | 用户能看懂同步状态与体积 | 设置页可见 status / 字节 / 错误 |
| G5 | 导入导出与在线同步共用合并语义 | 一套 `mergeHomepage` |
| G6 | Safari 诚实降级 | 弱 sync 时提示 + 文件同步包兜底 |

### 1.2 非目标（v1）

- Chrome ↔ Safari 账号级自动互通  
- 实时协同编辑  
- 端到端加密  
- 同步上传图标像素、壁纸大图、备份列表、访问历史  
- 自建同步服务器（接口预留，实现后置）

### 1.3 成功标准

- 安全闸门单测全绿（空远端、丢 ID、半片拒绝）  
- 200 枚以内书签稳定同步，少误触硬关 `syncEnabled`  
- S1–S12 场景表（§8）手工或 mock 通过  
- popup / app 写路径共用 domain，无第二套 LWW

## 2. 设计取舍（吸收来源）

### 2.1 从 Pass 吸收（必做）

| Pass 经验 | Homepage 落地 |
|-----------|----------------|
| 字段/实体时钟 + 确定性破同分 | `updatedAt` + `deviceId` 字典序；空值不覆盖有值 |
| 远端空不覆盖本地 | 协议不变量 + 测试 |
| 合并后稳定 ID 保全 | merge 结果校验，失败整单拒绝 |
| 墓碑 / 永久删除 | `deletedAt` + 可选 `purgedAt` |
| 集合成员状态 | group 成员 / 树父级用 placement 状态，禁止裸数组整表盖 |
| 写前 safety 快照 | 同步 pull 写盘前强制备份槽 |
| Outbox + 指数退避 | `homepage_sync_outbox` |
| 乐观并发 | `meta.revision` + `contentHash` |
| 手动包 = 自动 merge | `homepage.sync.bundle.v1` |
| 策略常量单点 | `sync_policy.js` |
| 诊断面板 | 设置页 status UI |

### 2.2 从《同步设计参考》吸收（选做 / 后期）

| 点 | 阶段 |
|----|------|
| 稳定逻辑 ID（已有 itm/grp） | 已具备，固化规范 |
| Tombstone 防幽灵 | Phase 1 |
| Op log + HLC 严格时间序 | Phase 3（可选） |
| 事件幂等 eventId | Phase 3 |

### 2.3 明确不搬

- AES-GCM 同步密钥、服务端只存信封  
- HTTP ETag/412 服务器协议（用 revision 模拟即可）  
- Passkey/TOTP 领域模型  
- 默认「按 URL 自动当成同一书签」

## 3. 现状与改造点

### 3.1 现状流水线

```
persistData → saveData(sync?) → sanitizeForSync → storage.sync 单 key
init/onChanged → local + sync 比 lastUpdated → 整包替换
```

### 3.2 主要缺陷

1. 整包 LWW：并发编辑互踩  
2. 单条 ~8KB：书签稍多即 quota 并关同步  
3. app / popup 双路径  
4. 无 revision、无 outbox、无同步专用 safety  
5. 成员数组整表覆盖  

### 3.3 目标流水线

```
UI/domain mutation
  → touch 实体时钟
  → save local homepage_data（必须先成功）
  → 若 syncEnabled：enqueue outbox / schedulePush
       → toProjection → pack shards
       → 读 meta.revision，冲突则 pull-merge 再推
       → 写 shards → 写 meta（revision++）

init / onChanged(sync)
  → pull shards（不齐则中止）
  → mergeHomepage(local, remote) + 安全闸门
  → 写前 safety 快照 → 写 local → render
  → 若本地更新则 schedulePush
```

## 5. 数据模型

### 5.1 设备与文档身份

```ts
// local: homepage_device_id
{ deviceId: string }  // UUID，安装时生成，永不改

// 写入 local homepage_data.settings 或独立 homepage_sync_state
{
  enabled: boolean,
  docId: string | null,       // 逻辑文档；首次成功绑定后固定
  status: "off" | "idle" | "syncing" | "quota" | "error" | "need_setup",
  lastPullAt: number,
  lastPushAt: number,
  lastError: string,
  lastRemoteRevision: number,
  bytesEstimate: number
}
```

### 5.2 本地完整状态（兼容现结构，增量字段）

在现有 `Node` / `Group` 上增加（缺省迁移补 0 / 当前时间）：

```ts
// Node 增量
{
  updatedAt: number,          // 已有则沿用
  updatedBy?: string,         // deviceId
  deletedAt?: number,         // 墓碑
  purgedAt?: number,          // 永久删除，拒绝旧数据复活
  titleUpdatedAt?: number,    // Phase 1b 字段级；Phase 1a 可只用 updatedAt
  urlUpdatedAt?: number
}

// Group 增量
{
  updatedAt: number,
  updatedBy?: string,
  deletedAt?: number,
  purgedAt?: number,
  nameUpdatedAt?: number,
  // members 不再只信 group.nodes 裸数组为同步真相：
  // 同步投影使用 placements；本地 group.nodes 仍是物化视图
}

// 可选 settings 时钟
settingsClocks?: Record<string, { at: number, by: string }>
```

### 5.3 同步投影 `SyncDocument`（进 storage.sync）

```ts
type SyncDocument = {
  schema: "homepage.sync.doc.v1",
  schemaVersion: 1,
  docId: string,
  revision: number,           // 乐观锁
  deviceId: string,           // 最后写入设备
  writtenAt: number,
  contentHash: string,        // 规范化 JSON hash

  settings: Record<string, unknown>,      // 白名单字段 only
  settingsClocks?: Record<string, { at: number, by: string }>,

  groups: SyncGroup[],
  nodes: SyncNode[],
  placements: SyncPlacement[]  // 树/分组成员关系
}

type SyncGroup = {
  id: string
  name: string
  order: number
  updatedAt: number
  updatedBy: string
  deletedAt?: number
  purgedAt?: number
}

type SyncNode = {
  id: string
  type: "item" | "folder"
  title: string
  url?: string
  iconType: "auto" | "color" | "remote" | "letter"
  iconData?: string        // 仅 remote URL 或极短；禁止大 dataURL
  color?: string
  updatedAt: number
  updatedBy: string
  deletedAt?: number
  purgedAt?: number
  titleUpdatedAt?: number
  urlUpdatedAt?: number
}

// 一个 node 在某一 parent 下的位置（group 或 folder）
type SyncPlacement = {
  nodeId: string
  parentKind: "group" | "folder"
  parentId: string
  index: number
  updatedAt: number
  updatedBy: string
  deletedAt?: number       // 从该 parent 移除
}
```

### 5.4 设置同步白名单

**同步：**  
`language, openMode, showSearch, searchEngineUrl, enableSearchEngine, gridDensity, fixedLayout, fixedCols, fontSize, theme, tooltipEnabled, keyboardNav, defaultGroupMode, defaultGroupId, sidebarCollapsed, sidebarHidden, iconFetch, iconRetryAtSix, iconRetryHour, backgroundType, backgroundColor, backgroundGradient*, backgroundOverlayStrength, maxBackups, emptyHintDisabled`

**不同步：**  
`lastSaveUrl, lastSaveTs, lastSaveToast, lastActiveGroupId（可选：可同步）, backgroundCustom, sync 过程临时字段`

> `lastActiveGroupId`：默认**不同步**（各设备记住自己上次分组更自然）。

### 5.5 同步包文件（手动 / Safari 桥）

```json
{
  "schema": "homepage.sync.bundle.v1",
  "exportedAtMs": 0,
  "source": {
    "app": "homepage-extension",
    "platform": "chrome|firefox|safari",
    "deviceId": "...",
    "formatVersion": 1
  },
  "payload": { /* SyncDocument 或与之等价的 groups/nodes/placements/settings */ }
}
```

导入：归一化 → `mergeHomepage` → safety → 写 local → 若开启同步则 schedulePush。

## 6. 合并语义（核心）

### 6.1 入口

```ts
function mergeHomepage(
  local: HomepageData,
  remote: SyncDocument,
  opts: { deviceId: string, now: number }
): { ok: true, state: HomepageData, stats: MergeStats }
  | { ok: false, reason: string }
```

内部：`local → SyncDocument` 投影后与 remote 合并，再物化回 `HomepageData`。

### 6.2 安全闸门（Pass 级不变量）

| # | 规则 |
|---|------|
| S1 | remote 无有效 groups/nodes（空文档）且 local 非空 → **不应用**，返回 ok 保持 local 或 reason=`empty_remote` |
| S2 | 合并结果缺少任何「local 中未 tombstone 的 id」→ **拒绝** `missing_local_ids` |
| S3 | shards 不齐 / hash 不匹配 → **拒绝** `incomplete_remote` |
| S4 | remote.purgedAt 存在的 id，local 不得被更旧数据复活 |
| S5 | 写盘前必须写入 safety 快照（失败则中止应用） |

### 6.3 字段裁决 `newerField`（移植 Pass 思想）

```
1. 比 fieldUpdatedAt（或实体 updatedAt）
2. 相等：空字符串不覆盖非空
3. 比 deviceId 字典序
4. 比值的 localeCompare
```

### 6.4 实体规则

| 实体 | 规则 |
|------|------|
| Node | 同 id：purged 优先；否则 deletedAt 与 updatedAt 按时钟比；存活则字段级或整实体 LWW（v1 允许整实体 LWW + 空值保护） |
| Group | 同 id：name/order 字段时钟；deleted/purged 同 node |
| Placement | 同 (nodeId, parentKind, parentId)：updatedAt 大者胜；deletedAt 表示移出 |
| Settings | 白名单 key 各自时钟；无时钟则用 doc.writtenAt |
| 仅一侧存在 | 并入（除非被对侧 purged） |

### 6.5 物化

1. 收集未 deleted/purged 的 groups/nodes  
2. 按 placements 生成 `group.nodes` / `folder.children`（按 index 排序）  
3. `dedupeData` + 孤儿 GC（注意：tombstone id 不进 nodes 表或进但不可见）  
4. 更新 `lastUpdated = now`（兼容旧字段，**不再作为唯一合并依据**）

## 7. 传输层（storage.sync）

### 7.1 分片

```
homepage_sync_meta = {
  schema: "homepage.sync.meta.v1",
  docId, revision, contentHash,
  shardCount, deviceId, writtenAt,
  bytesTotal
}
homepage_sync_s0 … s{N-1}  // 每片 UTF-8 ≤ SYNC_SHARD_MAX_BYTES (7000)
```

- groups + settings 放 s0 头部或 meta 侧车  
- nodes / placements 按数组切片  
- **先写 shards，最后写 meta**；读时 meta 与 shardCount/hash 不一致则丢弃  

### 7.2 配额策略（sync_policy）

```js
export const SYNC_SHARD_MAX_BYTES = 7000
export const SYNC_TOTAL_SOFT_BYTES = 60_000
export const SYNC_TOTAL_HARD_BYTES = 90_000
export const SYNC_TOMBSTONE_TTL_MS = 60 * 24 * 3600 * 1000
export const SYNC_OUTBOX_MAX_ATTEMPTS = 12
export const SYNC_OUTBOX_BASE_DELAY_MS = 5000
export const SYNC_OUTBOX_MAX_DELAY_MS = 3600_000
export const SYNC_PUSH_DEBOUNCE_MS = 1000
export const SYNC_PULL_DEBOUNCE_MS = 400
export const SYNC_REVISION_RETRY = 3
export const SYNC_SAFETY_MAX = 5
```

| 级别 | 行为 |
|------|------|
| 绿 | 正常 push |
| 黄 | UI 警告 |
| 红 / 写失败 | `status=quota` 或 `error`；**不**自动 `syncEnabled=false`；pending 留 outbox |

投影内自动裁剪（不删用户可见书签）：截断 title、去掉 remote 大 iconData、GC 过期 tombstone。

### 7.3 Outbox

```ts
// local homepage_sync_outbox
[{
  targetKey: "storage.sync|homepage",
  payload: SyncDocument,  // 或 contentHash 指向待推投影
  createdAtMs, attempts, nextRetryAtMs, lastError
}]
```

- 同 targetKey 只保留最新  
- 退避算法对齐 Pass `syncOutboxRetryDelayMs`  
- newtab init / background（若有）/ 手动「立即同步」触发 flush  

### 7.4 写序与并发

1. 变更 → 写 local  
2. 构造 projection  
3. `get meta`：若 `remote.revision > lastRemoteRevision` → pull merge 先  
4. `revision = remote.revision + 1`（无 remote 则 1）  
5. put shards + meta  
6. 若写后读回 hash 不符 → 重试有限次

## 8. 场景验收表（S1–S12）

| ID | 场景 | 期望 |
|----|------|------|
| S1 | A 增书签，B pull | B 可见 |
| S2 | A 改标题，B 改同卡 URL，时钟不同 | 各字段或整实体按时钟合并，不丢无关组内其他卡 |
| S3 | A、B 各增不同 id | 合并后都在 |
| S4 | A 删卡（tombstone），B 无更新 | 各端删除 |
| S5 | A 删，B 持旧包 upsert 更旧 | 保持删除 |
| S6 | A purged，B 旧活跃 | 不复活 |
| S7 | A 组内加卡，B 组内加卡 | placements 并集，两卡都在 |
| S8 | 空 remote | local 不变 |
| S9 | 缺 shard | 不应用 |
| S10 | 超配额 | local 完整，status=quota，outbox 重试 |
| S11 | 首次双端有数据 docId 不同 | UI 三选一，禁止静默 |
| S12 | popup 保存后 B 打开 | 可见新卡（domain 统一） |

## 11. 迁移

| 从 | 到 |
|----|----|
| sync 整包 `homepage_data` | 读出 → `toProjection` → 写 meta+shards；可选 delete 旧 key |
| 无 deviceId | 生成并持久化 |
| 无实体 deletedAt | `deletedAt=0` 表示存活 |
| schemaVersion 本地数据 | 保持 1；**同步协议**用 `homepage.sync.doc.v1` 独立 |

开启同步时：

1. 确保 deviceId  
2. 迁移旧 sync 整包（若有）  
3. 若 remote 空 → 上传 local 投影，docId=新  
4. 若 remote 有且 local 空 → 下载  
5. 若双有且 docId 冲突 → need_setup

## 19. 决策冻结（开发按此执行，变更需改本文）

1. 合并模型：**状态投影 + 实体/字段 LWW**（非 v1 全量 op log）  
2. 关系：**placements 状态**，非裸数组 LWW  
3. 删除：tombstone 60 天；支持 purged  
4. 超配额：**不关** `syncEnabled`，只标 `quota`  
5. 写序：**local 先于 sync**  
6. 空远端 / 丢 ID：**拒绝合并**  
7. app/popup：**单一 domain**  
8. 跨浏览器账号：**不做**；用同步包
